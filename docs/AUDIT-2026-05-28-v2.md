#### NH-1: Weak API keys in both .env files

**Files:** `app1-crypto-treasury/backend/.env`, `app2-fiat-settlement/.env`

```env
# App 1:
RELEASE_API_KEY=app2-secret-key-change-in-production

# App 2:
APP1_RELEASE_API_KEY=app2-secret-key-change-in-production
```

Both files use the same predictable key `app2-secret-key-change-in-production`. Since the repository is public, this key is effectively known to everyone. Anyone can call `POST /api/release` in App 1 with this key and release USDC.

**Fix:** Generate cryptographically strong keys:

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

Replace the values in both `.env` files.

#### NM-1: Middleware order in App 2 вЂ” helmet after static

**File:** `app2-fiat-settlement/backend/src/index.js`, lines 17вЂ“21

```js
app.use(express.json());
app.use(helmet());
app.use(cors(...));
app.use('/api/', rateLimit(...));
app.use(express.static(...));  // в†ђ static AFTER helmet вњ…
```

The order is correct. But `app.use(express.static(...))` comes after `helmet()`, which means helmet headers only apply to API routes. Static HTML files (dashboard) do not get `Content-Security-Policy`, `X-Frame-Options`, etc.

**Recommendation:** Declare helmet before static:

```js
app.use(helmet());
app.use(cors(...));
app.use('/api/', rateLimit(...));
app.use(express.static(path.join(__dirname, '../public'))); // static also gets helmet
```

This is almost already the case вЂ” `express.static` comes after `helmet`, which is correct. Just make sure this is intentional.

#### NL-1: confirmPayment in db.js does not handle 'processing' status

**File:** `app2-fiat-settlement/backend/src/db.js`, lines 109вЂ“117

```js
function confirmPayment(db, id, { app1_released }) {
  if (!getPayment(db, id)) throw new Error('Payment not found');
  db.prepare(`
    UPDATE payments
    SET status = 'sent', sent_at = datetime('now'), app1_released = ?
    WHERE id = ?
  `).run(app1_released ? 1 : 0, id);
  return getPayment(db, id);
}
```

After the atomic update in `index.js`, the payment has `processing` status. Then `confirmPayment` updates it to `sent` without a check. This is correct вЂ” but there is no guard against calling `confirmPayment` on an already `sent` payment directly (not via API). Not critical in current code, but worth adding a function-level check for robustness.
