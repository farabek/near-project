#### NC-1: App 2 .env не содержит ADMIN_API_KEY — приложение не запустится

**Файл:** `app2-fiat-settlement/.env`

```env
PORT=3001
APP1_URL=http://localhost:3000
APP1_RELEASE_API_KEY=app2-secret-key-change-in-production
DB_PATH=./data/app2.db
# ADMIN_API_KEY отсутствует!
```

`validateConfig()` теперь требует `ADMIN_API_KEY`, но фактический `.env` его не содержит. При запуске `node src/index.js` сервер немедленно упадёт с ошибкой:

```
Failed to initialize App 2: ADMIN_API_KEY is required. Check your .env file.
```

**Исправление:** Добавить в `app2-fiat-settlement/.env`:
```env
ADMIN_API_KEY=your_strong_admin_key_here
```

#### NH-1: Слабые API-ключи в обоих .env файлах

**Файлы:** `app1-crypto-treasury/backend/.env`, `app2-fiat-settlement/.env`

```env
# App 1:
RELEASE_API_KEY=app2-secret-key-change-in-production

# App 2:
APP1_RELEASE_API_KEY=app2-secret-key-change-in-production
```

Оба файла используют одинаковый предсказуемый ключ `app2-secret-key-change-in-production`. Поскольку репозиторий публичный, этот ключ де-факто известен всем. Любой может вызвать `POST /api/release` в App 1 с этим ключом и разблокировать USDC.

**Исправление:** Сгенерировать криптографически стойкие ключи:
```bash
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```
Заменить значения в обоих `.env`.

### 🟡 СРЕДНИЕ

---

#### NM-1: Порядок middleware в App 2 — helmet после static

**Файл:** `app2-fiat-settlement/backend/src/index.js`, строки 17–21

```js
app.use(express.json());
app.use(helmet());
app.use(cors(...));
app.use('/api/', rateLimit(...));
app.use(express.static(...));  // ← static ПОСЛЕ helmet ✅
```

Порядок корректный. Но `app.use(express.static(...))` идёт после `helmet()`, что значит helmet-заголовки применяются только к API-роутам. Статические HTML-файлы (дашборд) не получают заголовков `Content-Security-Policy`, `X-Frame-Options` и т.д.

**Рекомендация:** Переставить `express.static` перед роутами, но объявить helmet перед ним:
```js
app.use(helmet());
app.use(cors(...));
app.use('/api/', rateLimit(...));
app.use(express.static(path.join(__dirname, '../public'))); // статика тоже получает helmet
```
Это уже почти так — `express.static` стоит после `helmet`, что правильно. Просто убедитесь что это осознанно.

#### NM-2: Статус 'processing' не обрабатывается на фронтенде

**Файл:** `app2-fiat-settlement/backend/public/payments.html`

Новый статус `processing` (добавлен для защиты от race condition) не имеет CSS-класса и отображается без стиля, что сбивает пользователя.

```js
.badge.sent { background: #dcfce7; color: #16a34a; }
.badge.pending { background: #dbeafe; color: #1d4ed8; }
// .badge.processing — отсутствует!
```

**Рекомендация:** Добавить стиль в CSS и кнопку "Отправить" не должна показываться для `processing`:
```css
.badge.processing { background: #fef9c3; color: #a16207; }
```
```js
<td>${['pending', 'processing'].includes(p.status) && p.status === 'pending'
  ? `<button ...>✓ Отправить</button>` : ''}</td>
```

#### NM-3: /api/payments в App 1 не защищён API-ключом

**Файл:** `app1-crypto-treasury/backend/src/index.js`

```js
app.get('/api/payments', async (req, res) => { ... });      // ← без requireApiKey
app.get('/api/payments/:id', async (req, res) => { ... });  // ← без requireApiKey
```

`GET /api/balance`, `GET /api/payments`, `GET /api/payments/:id` доступны без ключа. Это означает, что историю всех escrow-транзакций может посмотреть кто угодно.

Для testnet это приемлемо, но стоит осознанно решить: должны ли read-only эндпоинты быть публичными или тоже защищёнными.

#### NL-1: confirmPayment в db.js не обрабатывает статус 'processing'

**Файл:** `app2-fiat-settlement/backend/src/db.js`, строки 109–117

```js
function confirmPayment(db, id, { app1_released }) {
  if (!getPayment(db, id)) throw new Error('Payment not found');
  db.prepare(`
    UPDATE payments
    SET status = 'sent', sent_at = datetime('now'), app1_released = ?
    WHERE id = ?
  `).run(app1_released ? 1 : 0, id);
  return getPayment(db, id);
}
```

После атомарного обновления в `index.js` платёж имеет статус `processing`. Затем `confirmPayment` обновляет его на `sent` без проверки. Это корректно — но нет защиты от вызова `confirmPayment` на уже `sent` платеже напрямую (не через API). В текущем коде это не критично, но стоит добавить проверку на уровне функции для надёжности.
